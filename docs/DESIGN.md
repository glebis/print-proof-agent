# PrintProof — агент-корректор типографии

**Дата:** 2026-06-06 · **Статус:** реализовано (v1, пилот) · **Код:** `apps/print-proof-agent/`
**Происхождение:** JTBD-интервью → аудит Codex → SVG-прототипы → TDD-реализация (одна сессия, лаборатория 05)

## 1. Job to Be Done

Менеджер типографии: срочные заказы упираются в корректуру (~неделя при цели «день») — поиск свободного корректора, ожидание правок, переговоры о сроках и цене через почту/мессенджеры/Excel.

> «Когда заказ доходит до корректуры, я хочу получить вычитанный текст без уговоров и переговоров с людьми — чтобы это занимало день вместо недели, а срок и цена были известны заранее».

- **Амбиция:** исключить внешних корректоров на малых формах (буклеты, листовки, финальная вёрстка); остаются менеджеры.
- **Switch forces:** push — корректоры узкое место («идём у них на поводу», водители за правками); pull — день вместо недели, предсказуемые срок/цена, отчёт с приёмкой; habit — часть клиентов требует живого корректора (эти заказы вне зоны агента); anxiety — репутационный риск ошибки в тираже.
- Полный JTBD-бандл: `~/jtbd/print-proof-agent/` (если сохранён) и в истории сессии.

## 2. UX-принцип (главный)

**Оптимизация под человека-проверяющего; агент работает автономно.**
1. Каждая правка несёт provenance: модуль (`rule-pass` / `llm-pass` / `vision-pass` / `profile`), основание, источник (стр./абзац, запись профиля), уверенность.
2. Каждое действие пользователя объясняет последствия до клика (тултипы: «6 правок → accepted, запись в протокол»).
3. Всё логируется в протокол приёмки (кто/что/когда).

Сохранено в память проекта: `project_printproof_ux_principle.md`.

## 3. Архитектура: B-lite (каскад)

Выбор по аудиту Codex (вместо «оркестратор + параллельные субагенты по категориям»): **категории живут в данных и UI, а не в runtime-субагентах** — поле `category`, не отдельные агенты. Причины: дедуп/конфликты пересекающихся правок, стоимость ~3–4×, латентность демо, «ложное соответствие UX и архитектуры».

```
Загрузка → Нормализация (immutable rev.1) → Оценка (мгновенно)
  → Rule-pass (TS, 0 токенов): тире, ёлочки, пробелы, словари клиента
  → LLM-pass (один query() Agent SDK): орфография/пунктуация/стиль → Edit[]
  → Vision-pass (только PDF/PNG, Agent SDK + Read): макет → LayoutIssue[] → задание дизайнеру
  → Resolver (код): дедуп, конфликты по span, сортировка
  → Приёмка менеджером (поштучно / категорией / все) → Экспорт MD/DOCX + Протокол
```

Ключевые инженерные решения:
- **Offsets:** все span'ы — в координатах immutable-текста одной ревизии; экспорт собирает текст курсором по отсортированным accepted-правкам. НЕ применять правки по одной с пересчётом (классический баг; принцип как в CRDT/git).
- **LLM не считает offsets:** llm-pass возвращает `original` + `context_before`, позицию находит код (`parseAndLocate`); галлюцинированные фрагменты отбрасываются.
- **Vision без координат:** vision даёт `block_hint` («правый нижний угол»), не автоправки — точность пикселей ненадёжна (аудит Codex).
- **Оценка детерминированная:** формула, не LLM — одинаковый объём ⇒ одинаковая цена (предсказуемость = ценность из JTBD).
- **Каскад в отдельном процессе** (`scripts/run-check.ts` через spawn): SDK-вызов не блокирует event loop Next.js.
- **Идемпотентность:** `runCheck` чистит только `pending`-правки — решения менеджера переживают повторный прогон.
- **Подписка, не API-ключ:** `ANTHROPIC_API_KEY` удаляется из env перед вызовами SDK.

## 4. Модель данных (SQLite, с правками Codex)

- `orders`: id, filename, type (`text|layout|pdf`), status (`uploaded→estimated→checking→review→done|error`), estimate_json, client_profile_id, text_hash, document_revision, error, created/updated_at
- `edits`: span_start/end, base_revision, original, suggested, context_before/after, category, severity, confidence, reason, source_module, decision (`pending/accepted/rejected`), decided_at/by
- `layout_issues`: page, block_hint, description, category, severity, confidence, decision
- `client_profiles`: name, rules_json (ё/е, кавычки, бренд-словарь), notes
- `artifacts`: source | parsed_text | export_md | export_docx | protocol

## 5. UI (стиль Tufte, по SVG-прототипам в `demos/print-proof-agent/prototypes/`)

- **Заказы:** drag-drop, таблица «объём · время · цена», статусы, «Открыть приёмку →»
- **Приёмка (вариант A, split view):** слева документ с подчёркнутыми предложениями (#c45a28) и diff-превью; справа правки по категориям с «Принять/Отклонить», «принять категорию», «принять все»; прогресс-полоса; provenance-тултипы на всём
- Токены: фон `#fffff8`, EB Garamond, Monaspace Argon для цифр, тонкие линейки, без тёмных плашек

## 6. Guardrails

1. Экспорт заблокирован (HTTP 409), пока есть правки без решения — проверено e2e.
2. Применяются только `accepted`; `original` сверяется с base-текстом (защита ревизий), пересечения → ошибка.
3. Ничего не отправляется автоматически; клиенты, требующие живого корректора, — вне зоны агента.
4. Протокол приёмки экспортируется отдельным форматом.

## 7. Тестирование

- **TDD, 17 unit-тестов:** rules (5), resolver (4), export (4), proofread-parse (4) — `npm test`
- **E2E через browser-mate:** загрузка → каскад → приёмка кликами → guardrail 409 → принять все → MD/DOCX/Протокол валидны
- **Vision-pass на реальном PNG:** 5 замечаний (кавычки, контраст, кегль, обрезанный текст)

## 8. Зафиксированные ограничения v1 / план v2

- DOCX-экспорт без сохранения оригинальной вёрстки (Markdown→docx, новое форматирование)
- При перегрузке API llm-pass может вернуть пусто → заказ только с rule-pass-правками; повтор: `npx tsx scripts/run-check.ts <id>`
- v2: OCR бумажных правок корректоров, точный DOCX-патчинг (mapping span→runs), почтовый канал заказов, многопользовательность
